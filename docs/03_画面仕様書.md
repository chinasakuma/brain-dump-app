# ブレインダンプアプリ 画面仕様書（バニラHTML版）

## 1. 文書概要

| 項目 | 内容 |
|------|------|
| アプリ名 | ブレインダンプ |
| 対象ページ | **単一画面**のみ（すべて `index.html` 内。認証 UI とメモ UI を切り替え表示） |
| 前提技術 | HTML / CSS / JavaScriptのみ。React・Vue・Next.js・npmは使わない。Supabase JS SDK は CDN 経由で読み込む。 |

**関連仕様：** データの項目・型・テーブル方針は **`docs/02_DB仕様書.md`** を参照すること。  
本書では **画面上の構成と振る舞い** に焦点を置く。**コード・SQL・接続情報（Supabase の URL／anon key 等）は本書には含めない。**

---

## 2. 画面一覧

このアプリでは **`index.html` 1 画面** のみを定義する。別 HTML ファイルによる画面分割は行わない。**未ログイン時は認証 UI**、**ログイン後はメモ UI** を同一ページ内で表示切替する。

| 表示モード | 概要 |
|------------|------|
| **認証 UI** | メール・パスワードによる **新規登録** と **ログイン** |
| **メモ UI** | メモの新規作成・一覧・編集・削除。**ログイン中のメール表示**と **ログアウト** |

---

## 3. `index.html`（認証 UI ＋ メモ UI）

### 3.1 画面名・目的

| 項目 | 内容 |
|------|------|
| **画面名** | ブレインダンプ（単一ページ） |
| **画面の目的** | 未ログイン時は認証、ログイン後は思考・アイデアのメモ保存・一覧・編集・削除まで完結させる。 |

### 3.2 レイアウトの大まかな構成（上から下）

共通：

1. アプリタイトル（ヘッダー）— タグラインは **認証時／メモ時** で文言を切り替えてよい

**認証 UI 表示時**（メモ UI は非表示）：

2. **ログインフォーム**（メール・パスワード・ログインボタン）
3. **新規登録フォーム**（メール・パスワード・新規登録ボタン）

**メモ UI 表示時**（認証 UI は非表示）：

2. **認証表示バー** — ログイン中の **メールアドレス** と **ログアウト** ボタン
3. **入力フォーム領域**（タイトル・本文・**カテゴリ（任意）**・保存ボタン）
4. **メッセージ表示エリア**（成功・エラー等の短文）
5. **メモ一覧エリア**

認証のエラーは **`alert()`**（メッセージ表示エリアはメモ操作用）。

### 3.3 認証 UI の要素

| 要素名 | 役割 |
|--------|------|
| **メール入力（ログイン用）** | ログイン時のメールアドレス |
| **パスワード入力（ログイン用）** | ログイン時のパスワード |
| **ログインボタン** | ログイン成功後 **メモ UI を表示** |
| **メール入力（登録用）** | 新規登録時のメールアドレス |
| **パスワード入力（登録用）** | 新規登録時のパスワード（6文字以上） |
| **新規登録ボタン** | 登録成功後 **メモ UI を表示** |

### 3.4 メモ UI の要素と各要素の役割

テーブルのカラム名（`title`, `content`, `category_id` など）は **`docs/02_DB仕様書.md`** に合わせる。

| 要素名 | 役割 |
|--------|------|
| **アプリタイトル** | 「ブレインダンプ」など、このアプリであることが一目で分かる文言を表示する。 |
| **ログイン中メール表示** | 現在のセッションの **メールアドレス** を表示する（例：「ログイン中: user@example.com」）。 |
| **ログアウトボタン** | ログアウト機能を実行し、**認証 UI を表示**する（メモ UI は非表示）。 |
| **タイトル入力欄** | 新規または編集中のメモの **タイトル** を入力する。DB の `title` に対応。 |
| **本文入力欄** | メモの **本文** を複数行で入力する。DB の `content` に対応。 |
| **カテゴリ選択（ドロップダウン）** | メモに紐づける **カテゴリ** を、マスタ `memo_categories` から読み込んだ選択肢から選ぶ。**未選択でもよい**。DB の `category_id`（NULL 可）に対応。 |
| **保存ボタン** | **新規保存** と **編集内容の更新** の両方に使う。**編集中どうかはアプリ内部の状態**（例：`editingId` の有無）で判定する。文言は「保存」など一つに統一してよい。 |
| **メモ一覧エリア** | DB から読み込んだメモを **一覧** で並べて表示する。並び順は **`created_at` が新しい順**（DB仕様・一覧機能仕様と一致）。 |
| **編集ボタン**（各メモ単位） | そのメモを編集対象にし、タイトル・本文・カテゴリ選択を入力欄に埋める。編集中メモの **id（UUID）** を内部で保持する。 |
| **削除ボタン**（各メモ単位） | そのメモの削除処理を開始する。実行前に **確認ダイアログ** を出す（後述「画面の状態」）。 |
| **メッセージ表示エリア** | 保存成功・更新成功・削除成功・入力チェック結果・通信エラー等を **短い文言** でユーザーに伝える。**しばらくしたら消す・次の操作で上書きする**など、見やすいルールでよい（詳細は機能仕様書）。 |

---

### 3.5 画面上の一覧で「見せる項目」（表示項目）

一覧の各行（またはカード）にユーザーが確認しやすいよう、最低限以下を表示する。**表示専用**の項目として整理する。

| 表示項目 | 内容・備考 |
|----------|-------------|
| **タイトル** | `title` の内容。**長い場合は省略表示**（複数行に折り返す／末尾を「…」にするなど、実装で調整）。 |
| **本文（概要）** | `content` の一部。**全文を常時表示すると長すぎる**ため、先頭のみ表示＋詳細は展開しない（今回は検索・折りたたみ機能は対象外。シンプルに数行または文字数で切ってよい）。 |
| **カテゴリ名** | メモが `memo_categories` に紐づく場合、その **カテゴリ表示名**。未設定の場合は「―」など短い占位表示でもよい（要求仕様 FR-07）。 |
| **作成日時** | `created_at` をユーザーが読みやすい形式で表示する（タイムゾーンはブラウザ表示に任せてよい）。 |

**内部利用のみ（画面上に見せても省略可）：** メモごとの **`id`（UUID）** は一覧に常に出す必要はないが、**編集・削除のリクエスト**には必ず使う。

---

### 3.6 入力項目一覧（メモ）

| 入力項目 | 画面上の名前（例） | 必須 | 備考 |
|----------|---------------------|------|------|
| タイトル | タイトル | 必須 | 空白のみは不可。**前後の余分な空白**は送信前に整える余地あり（機能仕様で定義）。 |
| 本文 | 本文 | 必須 | 空白のみは不可。複数行。 |
| カテゴリ | カテゴリ（ドロップダウン） | **任意** | 先頭は「未選択」（値なし）。選択肢本文は **`memo_categories` を DB から取得**した結果で構成する（要求仕様 FR-07）。 |

編集開始時は、選択したメモの `title` / `content` と、紐づく `category_id` に応じたドロップダウンの選択状態を入力欄に自動で反映する。

### 3.7 ボタン一覧

| ボタン | 場所 | 主な動作（概要） |
|--------|------|-------------------|
| **ログイン** | 認証 UI | ログイン後メモ UI 表示。**機能仕様書 §4.8** |
| **新規登録** | 認証 UI | 登録後メモ UI 表示。**機能仕様書 §4.7** |
| **ログアウト** | 認証表示バー | **signOut** 後認証 UI 表示。**機能仕様書 §4.9** |
| **保存** | 入力フォーム付近 | 新規：**INSERT**。編集中：**UPDATE**。処理後は入力欄をクリアし一覧を最新化。**詳細は機能仕様書**。 |
| **編集** | 各メモ | 入力欄にそのメモの内容を表示し、編集モードへ。 |
| **削除** | 各メモ | 確認後に **DELETE**、一覧更新。**詳細は機能仕様書**。 |

ブラウザ標準の **確認ダイアログ**（`confirm` 等）は「削除確認」状態で使用してよい。独自モーダルは必須としない。

### 3.8 画面の状態

| 状態名 | いつになるか | 画面上の見え方・ユーザー操作 |
|--------|----------------|------------------------------|
| **未認証（認証 UI）** | セッションが無い | ログイン・新規登録フォームのみ。**メモ UI は非表示** |
| **認証済み・初期表示（メモ UI）** | ログイン済みでページを開いた／ログイン直後 | メール表示・ログアウト・メモ入力・一覧。カテゴリ・一覧の読み込み開始 |
| **入力中** | ユーザーがタイトルまたは本文を編集しているとき | 通常の入力可能な見た目。**保存ボタンは押せる**（バリデーションは押下時）。 |
| **一覧表示** | 一覧取得成功後（通常時） | `created_at` 新しい順にメモが並ぶ。各メモに編集／削除。**編集中でなければ**入力欄は空または未編集状態。 |
| **編集中** | 編集ボタン押下後～保存またはキャンセル相当まで | 入力欄に選択メモのタイトル・本文が表示される。**保存ボタンは更新用**として振る舞うことをユーザーが迷わないよう、文言や表示で「編集中」を示すとよい（例：一覧の対象をハイライト、「編集中」の小見出し）。**キャンセル専用ボタンは今回の必須要件ではない**が、入力欄を空にして編集状態をやめる運用があれば明記した実装でもよい。 |
| **保存成功** | 新規 INSERT 成功直後 | 成功メッセージをメッセージ表示エリアに出す。**入力欄クリア**。一覧を最新表示。 |
| **保存失敗** | INSERT 失敗など | **エラー内容が分かる短い文言**。**入力内容はできるだけ残す**（ユーザーが再試行しやすい）。 |
| **更新成功** | UPDATE 成功直後 | 成功メッセージ。**編集状態解除**、入力欄クリア、一覧再表示。 |
| **削除確認** | 削除ボタン押下後、OK を待っている間 | ブラウザの確認ダイアログが開いている。キャンセルなら一覧はそのまま。 |
| **削除成功** | DELETE 成功後 | 成功メッセージ。**一覧から該当行が消える**（再取得で反映）。 |
| **エラー発生時** | 一覧取得失敗／通信失敗／想定外のレスポンス等 | メッセージ表示エリアに **失敗や再試行を促す文**。**可能なら**入力内容や一覧キャッシュの扱いは機能仕様書に準拠。 |

**状態の重なり：** 「一覧表示」のままメッセージだけが変わる（成功・エラー）は一般的。

### 3.9 スマホ表示で注意すること

| 観点 | 内容 |
|------|------|
| **横幅** | 画面幅が狭いとき、横スクロールが常に必要にならないよう、パディングと折り返しを調整する。 |
| **タップ領域** | 編集・削除・保存は **十分なサイズ・間隔** を確保し、誤タップを減らす。 |
| **本文入力** | テキストエリアは高さが低すぎないようにし、入力しやすくする（必要なら `min-height` 等）。 |
| **一覧の長さ** | メモが多いとき、ページ全体が読みやすいよう一覧エリアだけ縦スクロールしやすくする。**無限スクロールは対象外**でよいが、一覧が画面を圧迫しすぎない工夫は推奨。 |
| **文字サイズ** | 読みやすい最小サイズを確保。**端末設定のフォントサイズ**と相性が悪くなりすぎないよう過度に小さくしすぎない。 |

### 3.10 今回作らない画面・機能

以下は **仕様対象外** とする。

| 区分 | 対象外の内容 |
|------|----------------|
| 認証の拡張 | OAuth、マジックリンク、メール確認、パスワードリセット画面 |
| RLS のみの DB 強化 | `brain_dumps` への RLS 有効化（カラム追加以外の DB 変更。方針は DB 仕様書 §6） |
| データの構造化・探索 | **カテゴリマスタの画面からの編集**（一覧は読み込みのみ）、タグ機能、検索機能、AI要約機能 |
| メディア・アプリ構成 | 画像アップロード、Next.js化、React化、外部API連携（Supabase を除く） |
| 運用・文書ルールの一部 | 本番公開、`/.mdc` ルールファイルの作成や管理 |

詳細リストは機能仕様書の「今回作らないもの」と揃える。
