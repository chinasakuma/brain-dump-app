# ブレインダンプアプリ DB仕様書（Supabase）

## 1. 文書概要

本書は、Supabase（PostgreSQL）上で稼働させるブレインダンプアプリの**テーブル定義**およびデータアクセスに関する DB 側の仕様を定義する。

## 2. スキーマ方針

- アプリ固有のデータは **`public.brain_dumps`（メモ）** と、カテゴリ選択肢用の **`public.memo_categories`（マスタ）** の 2 テーブルで管理する。
- **`user_id` 列は設けない**。行単位での認証ユーザー紐付け・RLS によるユーザー別分離は行わない（単一利用者／プロジェクト内共有の運用前提。厳密なマルチテナントが必要になった際は別途 `user_id` の導入などを検討する）。
- メモ側の **`category_id` は NULL 可**であり、カテゴリ未設定のメモを許容する（要求仕様 FR-07）。
- Row Level Security（RLS）は**当面オフ**とする（有効化する場合は方針に合わせてポリシーを別途設計すること）。

## 3. テーブル一覧

| スキーマ | テーブル名 | 説明 |
|----------|------------|------|
| public | memo_categories | カテゴリのマスタ。ドロップダウン選択肢の元データ |
| public | brain_dumps | ブレインダンプ（メモ）1 件につき 1 行 |

## 4. テーブル定義: `public.memo_categories`

### 4.1 カラム

| カラム名 | 型 | 制約 | 説明 |
|----------|-----|------|------|
| id | uuid | PRIMARY KEY, `DEFAULT gen_random_uuid()` | カテゴリの識別子。自動採番 |
| name | text | NOT NULL, UNIQUE | カテゴリ表示名（ドロップダウン・一覧に表示） |
| sort_order | integer | NOT NULL, `DEFAULT 0` | ドロップダウン等の並び順（昇順で利用想定） |

### 4.2 主キー・外部キー

- **主キー:** `id`
- **外部キー:** なし（マスタのため参照元のみ）

### 4.3 推奨インデックス

| インデックス | 目的 |
|--------------|------|
| `CREATE UNIQUE INDEX ... ON public.memo_categories (name)` | `UNIQUE` 制約に相当（DDL ではテーブル定義に含めてもよい） |
| `CREATE INDEX ON public.memo_categories (sort_order)` | マスタ全件取得時の並び替え（件数が少ない場合は任意） |

## 5. テーブル定義: `public.brain_dumps`

### 5.1 カラム

| カラム名 | 型 | 制約 | 説明 |
|----------|-----|------|------|
| id | uuid | PRIMARY KEY, `DEFAULT gen_random_uuid()` | レコードの識別子。自動採番 |
| title | text | NOT NULL | タイトル（必須） |
| content | text | NOT NULL | 本文（必須） |
| category_id | uuid | **NULL 可**, FK → `memo_categories(id)` | 紐づくカテゴリ。未設定時は NULL |
| created_at | timestamptz | NOT NULL, `DEFAULT now()` | 作成日時（自動セット） |

**補足（編集要件との整合）**

- 要求仕様に「編集」がある場合、**更新日時**を保持するなら任意で `updated_at timestamptz` を追加する（本仕様の必須カラムではない）。

### 5.2 主キー・外部キー

- **主キー:** `id`
- **外部キー:** `category_id` → `public.memo_categories(id)`
  - **`ON DELETE SET NULL`** とする（マスタ行削除時、メモ行は残しカテゴリ紐づけのみ外す）。

### 5.3 推奨インデックス

| インデックス | 目的 |
|--------------|------|
| `CREATE INDEX ON public.brain_dumps (created_at DESC)` | 新しい順の一覧表示 |
| `CREATE INDEX ON public.brain_dumps (category_id)` | カテゴリで絞り込む・`memo_categories` と結合する場合 |

## 6. Row Level Security（RLS）

- 現状は **RLS を有効にしない**（`sql/create_brain_dumps.sql` および `sql/20260523_01_カテゴリマスタとメモへのFK追加.sql` の方針に合わせる）。
- 将来 RLS を有効にする場合、`user_id` がない場合は「認証済みならテーブル全体に CRUD」のような広いポリシーになり、**アカウント単位での行分離にはならない**点に注意すること。

## 7. アプリケーションからの利用（概要）

- クライアントは Supabase 経由で `brain_dumps` および `memo_categories` にアクセスする。
- 一覧でカテゴリ名を出す場合は、`brain_dumps` と `memo_categories` を **`category_id` で LEFT JOIN**（または PostgREST の `select` でリレーション指定）し、**`category_id` が NULL のメモ**はカテゴリ名なしとして扱う。
- **anon / service_role キーの取り扱い**は Supabase の推奨に従い、ブラウザに service_role は埋め込まない。
- メモデータの機密性を高めるには、適切な RLS／認証設計または別レイヤでの制御が必要である。

## 8. DDL（参考）

リポジトリ内の実行用 SQL:

- **新規プロジェクト用（テーブル一式）:** `sql/create_brain_dumps.sql` — `memo_categories` の作成、`brain_dumps`（`category_id` 含む）、インデックスまで含む。
- **既存の `brain_dumps` がある環境への追加:** `sql/20260523_01_カテゴリマスタとメモへのFK追加.sql` — マスタ作成、`brain_dumps` への `category_id` 追加、インデックス、仮データ INSERT。
- **既存から `user_id` を削除する場合:** `sql/remove_user_id_from_brain_dumps.sql`
