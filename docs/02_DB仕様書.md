# ブレインダンプアプリ DB仕様書（Supabase）

## 1. 文書概要

本書は、Supabase（PostgreSQL）上で稼働させるブレインダンプアプリの**テーブル定義**およびデータアクセスに関する DB 側の仕様を定義する。

**関連要求：** 要求仕様 **FR-08**（ユーザーはログイン後、自分が作成したメモのみを参照・追加できる）。

## 2. スキーマ方針

- アプリ固有のデータは **`public.brain_dumps`（メモ）** と、カテゴリ選択肢用の **`public.memo_categories`（マスタ）** の 2 テーブルで管理する。
- **今回の DB 変更は `brain_dumps` への `user_id` カラム追加のみ**とする。`memo_categories` のスキーマ変更は行わない。
- メモ行は **`user_id`** で Supabase Auth のユーザー（`auth.users.id`）に紐づけ、**ログインユーザー本人の行のみ**をアプリから参照・追加・更新・削除する前提とする（絞り込みの具体は機能仕様書）。
- メモ側の **`category_id` は NULL 可**であり、カテゴリ未設定のメモを許容する（要求仕様 FR-07）。
- Row Level Security（RLS）は**本変更の必須範囲外**（既存方針どおり当面オフでもよい）。`user_id` 追加後に RLS を有効化する場合は、所有者一致のポリシーを**別途**設計・適用すること（§6 参照）。

## 3. テーブル一覧

| スキーマ | テーブル名 | 説明 |
|----------|------------|------|
| public | memo_categories | カテゴリのマスタ。ドロップダウン選択肢の元データ（全ユーザー共通） |
| public | brain_dumps | ブレインダンプ（メモ）1 件につき 1 行。**所有者は `user_id`** |

## 4. テーブル定義: `public.memo_categories`

（変更なし。v1.2 と同一）

### 4.1 カラム

| カラム名 | 型 | 制約 | 説明 |
|----------|-----|------|------|
| id | uuid | PRIMARY KEY, `DEFAULT gen_random_uuid()` | カテゴリの識別子。自動採番 |
| name | text | NOT NULL, UNIQUE | カテゴリ表示名（ドロップダウン・一覧に表示） |
| sort_order | integer | NOT NULL, `DEFAULT 0` | ドロップダウン等の並び順（昇順で利用想定） |

### 4.2 主キー・外部キー

- **主キー:** `id`
- **外部キー:** なし（マスタのため参照元のみ）

### 4.3 推奨インデックス

| インデックス | 目的 |
|--------------|------|
| `CREATE UNIQUE INDEX ... ON public.memo_categories (name)` | `UNIQUE` 制約に相当 |
| `CREATE INDEX ON public.memo_categories (sort_order)` | マスタ全件取得時の並び替え（件数が少ない場合は任意） |

## 5. テーブル定義: `public.brain_dumps`

### 5.1 カラム

| カラム名 | 型 | 制約 | 説明 |
|----------|-----|------|------|
| id | uuid | PRIMARY KEY, `DEFAULT gen_random_uuid()` | レコードの識別子。自動採番 |
| **user_id** | **uuid** | **NOT NULL** | **メモの所有者。Supabase Auth のユーザー ID（`auth.users.id`）** |
| title | text | NOT NULL | タイトル（必須） |
| content | text | NOT NULL | 本文（必須） |
| category_id | uuid | **NULL 可**, FK → `memo_categories(id)` | 紐づくカテゴリ。未設定時は NULL |
| created_at | timestamptz | NOT NULL, `DEFAULT now()` | 作成日時（自動セット） |

**`user_id` の意味**

- 新規 INSERT 時に、**ログイン中セッションのユーザー ID** をセットする（クライアントまたは DB トリガー。実装は機能仕様書）。
- 一覧・単件取得・UPDATE・DELETE では、**`user_id` = ログイン中ユーザー** の行だけを対象とする。

**補足（編集要件との整合）**

- 更新日時を保持するなら任意で `updated_at timestamptz` を追加してよい（本仕様の必須カラムではない）。

### 5.2 主キー・外部キー

- **主キー:** `id`
- **外部キー:**
  - `category_id` → `public.memo_categories(id)` — **`ON DELETE SET NULL`**
  - **`user_id` → `auth.users(id)`** — 推奨: **`ON DELETE CASCADE`**（Auth ユーザー削除時に当該ユーザーのメモ行も削除）。プロジェクト方針で CASCADE を避ける場合は **`RESTRICT`** 等を選び、運用で整合を取る。

### 5.3 推奨インデックス

| インデックス | 目的 |
|--------------|------|
| `CREATE INDEX ON public.brain_dumps (user_id, created_at DESC)` | **ログインユーザー別**の一覧（新しい順） |
| `CREATE INDEX ON public.brain_dumps (category_id)` | カテゴリ結合・参照 |

（既存環境に `created_at` のみのインデックスがある場合、上記複合インデックスへ置き換えまたは追加を検討してよい。）

### 5.4 既存データへの `user_id` 追加（マイグレーション方針）

本リポジトリの **DB 変更範囲はカラム追加のみ**とするが、既に `brain_dumps` に行がある場合は次を満たすこと。

1. `user_id uuid` を **NULL 可**で追加する  
2. 既存行の `user_id` を、運用で決めた所有者（例: 特定の Auth ユーザー ID）に **UPDATE で埋める**、または不要行を削除する  
3. 未設定行が無いことを確認したうえで **`NOT NULL` 制約**を付与する  
4. 外部キー・インデックスを追加する  

**新規プロジェクト**では最初から `user_id NOT NULL` で作成してよい。

## 6. Row Level Security（RLS）

- ポリシーの詳細（テーブル・操作・条件）は **`docs/05_RLS仕様書.md`** を正とする。
- **`brain_dumps`:** `authenticated` に対し、**SELECT / INSERT / UPDATE / DELETE** すべて **`user_id = auth.uid()`**。
- **`memo_categories`（categories）:** `authenticated` に **SELECT のみ**（全行）。**INSERT / UPDATE / DELETE はクライアントに許可しない**。`anon` は両テーブルとも拒否。
- RLS 用 SQL の生成・適用は別タスク。スキーマ変更としては **`brain_dumps.user_id` 追加**が前提。

## 7. アプリケーションからの利用（概要）

- クライアントは Supabase 経由で `brain_dumps` および `memo_categories` にアクセスする。
- **メモの INSERT** には必ず **`user_id`**（セッションの `user.id`）を含める。
- **メモの SELECT / UPDATE / DELETE** には **`user_id` がログイン中ユーザーと一致する**条件を付ける（`id` のみでの操作は他ユーザー行に触れうるため不可）。
- 一覧でカテゴリ名を出す場合は、`brain_dumps` と `memo_categories` を **`category_id` で LEFT JOIN**（または PostgREST のリレーション指定）する。
- **anon / service_role キーの取り扱い**は Supabase の推奨に従い、ブラウザに service_role は埋め込まない。

リポジトリ内の実行用 SQL（**`user_id` 追加用の専用 SQL は実装フェーズで別途追加予定**。本書では仕様のみ定義）:

| 用途 | ファイル（参考） |
|------|------------------|
| 新規プロジェクト用（テーブル一式） | `sql/create_brain_dumps.sql` — **今後 `user_id` 含む形へ更新予定** |
| 既存環境へのカテゴリ追加 | `sql/20260523_01_カテゴリマスタとメモへのFK追加.sql` |
| anon アクセス権 | `sql/enable_anon_access.sql` |
| （廃止方向）`user_id` 削除 | `sql/remove_user_id_from_brain_dumps.sql` — **本要件では使用しない** |

**`user_id` 追加の参考 DDL（既存 `brain_dumps` がある場合）:**

```sql
-- 1. カラム追加（既存行があるときは一旦 NULL 可）
ALTER TABLE public.brain_dumps
  ADD COLUMN IF NOT EXISTS user_id uuid;

-- 2. 既存行の user_id を埋めたあと NOT NULL にする（値の設定は運用・移行手順に従う）
-- UPDATE public.brain_dumps SET user_id = '...' WHERE user_id IS NULL;

ALTER TABLE public.brain_dumps
  ALTER COLUMN user_id SET NOT NULL;

-- 3. 外部キー・インデックス
ALTER TABLE public.brain_dumps
  ADD CONSTRAINT brain_dumps_user_id_fkey
  FOREIGN KEY (user_id) REFERENCES auth.users(id) ON DELETE CASCADE;

CREATE INDEX IF NOT EXISTS brain_dumps_user_id_created_at_idx
  ON public.brain_dumps (user_id, created_at DESC);
```
