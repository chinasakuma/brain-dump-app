# ブレインダンプアプリ 機能仕様書（バニラHTML + Supabase）

## 1. 文書概要

| 項目 | 内容 |
|------|------|
| アプリ名 | ブレインダンプ |
| データ保存先 | Supabase PostgreSQL、`public.brain_dumps`（任意で `memo_categories` に FK）および **`public.memo_categories`**（マスタ） |
| クライアント | **`index.html` 1 ファイル**（認証・メモを同一 HTML 内で切り替え）。npmなし。**Supabase JavaScript SDK は CDN で読み込む**。 |

**関連仕様：** テーブル・カラムの定義は **`docs/02_DB仕様書.md`** を参照。  
本書では **画面上の機能と処理フロー、Supabase 操作の対応** を定義する。**コード・SQL・Supabase の URL／anon key 等は本書には書かない。**

## 2. 機能一覧

| No | 機能名 | 目的（一言） |
|----|--------|----------------|
| 1 | カテゴリ選択肢読込機能 | メモ入力用ドロップダウンを **`memo_categories` の内容** で埋める（`sort_order` 昇順など DB 仕様に合わせる） |
| 2 | メモ新規保存機能 | 入力したタイトル・本文および**任意の**カテゴリをデータベースに新規追加する |
| 3 | メモ一覧表示機能 | 保存済みメモを新しい順に画面に並べ、**カテゴリ名**も分かるように見せる |
| 4 | メモ編集機能 | 既存メモ（タイトル・本文・**カテゴリ任意**）を修正しデータベースに反映する |
| 5 | メモ削除機能 | 不要なメモをデータベースから削除する |
| 6 | メッセージ表示機能 | 成功・検証エラー・通信エラー等をユーザーに伝える（**認証エラーは対象外** — `alert()` を使用） |
| 7 | 新規登録機能 | メールアドレス・パスワードで Supabase Auth にユーザーを作成し、成功後 `index.html` へ遷移する |
| 8 | ログイン機能 | メールアドレス・パスワードでサインインし、成功後 `index.html` へ遷移する |
| 9 | ログアウト機能 | セッションを終了し、認証画面へ遷移する |
| 10 | ログイン中メール表示機能 | メモ画面で現在ログイン中のメールアドレスを表示する |

**認証の共通制約（要求仕様 FR-05 に準拠）：** OAuth・マジックリンク・メール確認（confirm email）は行わない。パスワードは **6 文字以上**。認証関連のエラーは **`alert()`** で通知する。

## 3. 共通の前提・用語

| 用語 | 説明 |
|------|------|
| **Supabaseクライアント** | CDN から読み込んだ SDK で初期化し、**Auth（登録・ログイン・ログアウト・セッション）** および `brain_dumps` / `memo_categories` にクエリするオブジェクト。**接続情報の設定は実装側で別途。** |
| **セッション** | ログイン成功後、ブラウザに保持される認証状態。メモ画面表示前に **有効なセッションがあること** を確認する。 |
| **ログイン中ユーザー ID** | セッションの `user.id`（UUID）。メモの **`brain_dumps.user_id`** およびクエリの絞り込みに用いる（要求仕様 FR-08・DB 仕様書）。 |
| **編集中フラグ／編集中 ID** | 編集対象メモの `id`（UUID）をブラウザ内で保持している状態。無ければ「新規」、あれば「更新」。 |
| **一覧の並び順** | `created_at` の **降順**（新しいものが先）。一覧表示機能と共通。 |

**メモ操作の共通ルール（FR-08・DB 仕様書に準拠）：**

- メモ関連の **INSERT / SELECT / UPDATE / DELETE** は、いずれも **ログイン中ユーザー ID** を前提とする。未ログインでは実行しない。
- **INSERT** 時は、ペイロードに **`user_id` = ログイン中ユーザー ID** を必ず含める。
- **SELECT（一覧・単件） / UPDATE / DELETE** 時は、条件に **`user_id` = ログイン中ユーザー ID** を必ず付与する（`id` のみの条件は不可）。
- 他ユーザーの `user_id` を持つ行は、一覧に表示せず、編集・削除の対象にもしない。

**入力チェックの共通ルール（推奨）：**

- **タイトル・本文とも必須**（DB が NOT NULL に合わせる）。
- 送信前に **前後の空白を除去（trim）** したうえで、空になったらエラーとしてメッセージ表示し **Supabase へは送信しない**。
- **カテゴリは任意**。ドロップダウン未選択時は **`category_id` を NULL** として送信し、一覧ではカテゴリ名なし（または占位表示）でよい。

長さの上限などは DB 側で明示的な制限が無い場合、**異常に大きな入力が遅くなる**点だけ留意（今回仕様での数値上限は別途未定でよい）。

## 4. 機能別仕様

### 4.1 カテゴリ選択肢読込機能

| 項目 | 内容 |
|------|------|
| **目的** | 投稿／編集フォームのドロップダウン選択肢を、**DB の `memo_categories`** で構成する |
| **Supabaseでの操作** | **SELECT** — `memo_categories` から **`id`, `name`（および並び順用の `sort_order`）** を取得し、**`sort_order` 昇順**など DB 仕様に従って並べる |

**処理の流れ**

1. **ページ表示後**に（一覧取得とは独立して実行してもよいが、入力前に済ませるとよい）  
2. `memo_categories` を **SELECT** する  
3. 取得結果を **ドロップダウンの option** として並べる。先頭に **値なし＝「未選択」相当** を置き、ユーザーが選ばなかった場合 **`category_id` は NULL** として保存する  

**正常時**

- メモの保存機能はカテゴリ未選択でも利用できること（要求仕様 FR-07）。  

**エラー時**

| 種別 | 推奨挙動 |
|------|----------|
| 読み込み失敗 | メッセージで通知。ドロップダウンを無効化するか、そのままでも **メモ一覧は可能なら表示** とするかは実装で統一するとよい |

---

### 4.2 メモ新規保存機能

| 項目 | 内容 |
|------|------|
| **目的** | ユーザーが入力した内容を **`brain_dumps` に新しい行として追加**し、一覧に反映させる |
| **Supabaseでの操作** | **INSERT（行の追加）** — `brain_dumps` に **`user_id`（ログイン中ユーザー ID）**, `title`, `content` を指定。**`category_id`** は選択時のみセットし、未選択なら **`NULL`**。**`id` と `created_at` はサーバ側既定**（DB 仕様に合わせる） |

**処理の流れ（正常系の目安）**

1. ユーザーがタイトルと本文を入力する（カテゴリは任意）
2. **編集中 ID が無いこと** を確認する（または保存時に無視して新規扱いになるよう統一する）  
3. ユーザーが **保存ボタン** を押す  
4. **入力チェック**（空白のみ不可）。NGなら メッセージ表示機能 でエラー文言を表示し終了  
5. ログイン中ユーザー ID を取得し、Supabase に **INSERT** を実行する（**`user_id`**, `title`, `content`, および任意の `category_id`）
6. 成功したら：**入力欄を空にする**（カテゴリも未選択に戻す）、**編集中状態を解除**（念のため）、**一覧を再読込**する、成功メッセージを表示する  

| 入力値 | 内容 |
|--------|------|
| 所有者 | セッションの **`user.id`** → DB の **`user_id`**（必須） |
| タイトル | 画面上のタイトル入力欄 → DB の `title` |
| 本文 | 画面上の本文入力欄 → DB の `content` |
| カテゴリ | 選択中の **`memo_categories.id`** → DB の `category_id`。未選択なら **`NULL`** |

| 出力・表示内容 | 内容 |
|----------------|------|
| 画面上のリスト | INSERT 後、一覧読み込みで **追加された行が先頭付近に表示**される（新しい順） |
| メッセージ | 例：「保存しました」など成功メッセージ |

**正常時の動き**

- データが1行増える  
- フォームが空になる  
- 一覧が最新になる  
- 成功メッセージが見える  

**エラー時の動き**

| 種別 | ユーザーへの伝え方 | アプリ側の推奨挙動 |
|------|--------------------|---------------------|
| 入力エラー（空など） | 「タイトルと本文を入力してください」等 | Supabase は呼ばない。入力内容は温存 |
| Supabase がエラーを返した | メッセージ表示エリアに **簡潔な失敗文言**（技術細語は多用しない） | 入力内容は温存。**再試行しやすい**状態にする |
| ネットワーク断など | 「接続に失敗しました」等 | 入力内容は温存 |

### 4.3 メモ一覧表示機能

| 項目 | 内容 |
|------|------|
| **目的** | 起動時および更新後に、**保存済みメモを一覧**で見せ、**カテゴリ名**が分かるようにする |
| **Supabaseでの操作** | **SELECT** — `brain_dumps` を **`user_id` = ログイン中ユーザー ID** で絞り込み、**`created_at` 降順**で取得。リレーションで **`memo_categories.name`** を同時に得る。**必要なカラム：** `id`, `user_id`, `title`, `content`, `created_at`, `category_id`, 表示用 **`memo_categories.name`**。 |

**処理の流れ**

1. **メモ UI 表示時**（および保存・更新・削除の成功後に「再読み込み」として同じ処理を実行）  
2. ログイン中ユーザー ID を取得し、**`user_id` 一致**を条件に Supabase へ一覧用 **SELECT** を実行する  
3. 取得結果を画面上の **メモ一覧エリア** に描画する（**カテゴリ未設定**は名前なしや「―」などで表示）  
4. 各メモに **編集ボタン** と **削除ボタン** を付ける  

| 入力値 | 内容 |
|--------|------|
| なし | 画面上の入力値は一覧取得には不要。**フィルタ・検索条件も今回つけない** |

| 出力・表示内容 | 内容 |
|----------------|------|
| 一覧項目 | **タイトル、本文の要約または冒頭、`created_at`、カテゴリ名**。各行に編集／削除。**`id` は画面に出さなくてもよい**が操作に必要 |

**正常時の動き**

- メモが0件なら一覧は空または「メモがありません」等の案内でもよい（文言は実装で調整）。  
- 1件以上なら指定順で表示。  

**エラー時の動き**

- メッセージ表示機能で「一覧を読み込めませんでした」等。**可能なら**再読込の手段（ページ再読み込みのみでも可）を示す。

### 4.4 メモ編集機能

| 項目 | 内容 |
|------|------|
| **目的** | 既存メモの `title` / `content`／**任意の `category_id`** を変えて **同じ `id` の行を更新**する |
| **Supabaseでの操作** | **UPDATE** — 条件: **`id` = 編集中 ID** かつ **`user_id` = ログイン中ユーザー ID**。更新項目: `title`, `content`, `category_id`（未選択時 `NULL`）。**`user_id` は変更しない**。 |

**処理の流れ**

1. ユーザーが某一覧項目の **編集ボタン** を押す  
2. そのメモの **`id`** を編集中として保持する  
3. サーバーから **`id` かつ `user_id` 一致**で当該行を **SELECT**（または一覧キャッシュから）し、`title`, `content`, `category_id` を入力欄に表示する。取得できない場合はエラー扱いとする  
4. ユーザーが内容を変更し **保存ボタン** を押す  
5. **入力チェック**。NGならメッセージのみ  
6. **`id` と `user_id` の両方**を条件に **UPDATE** を実行する（`title`, `content`, `category_id`）  
7. 成功したら：**編集中状態解除**、**入力欄を空にする**（カテゴリも未選択に戻す）、**一覧再読込**、成功メッセージ  

| 入力値 | 内容 |
|--------|------|
| 編集対象の id | `brain_dumps.id` |
| 更新後タイトル | `title` |
| 更新後本文 | `content` |
| カテゴリ | 選択中マスタ **`id`** → `category_id`、未選択なら **`NULL`** |

| 出力・表示内容 | 内容 |
|----------------|------|
| 一覧 | UPDATE 後に再取得した内容が反映されていること |

**正常時の動き**

- データベース上の該当行のみ内容が変わる（`created_at` はそのままでよい。**DB仕様上 `updated_at` は任意**）。  

**エラー時の動き**

- INSERT と同様、**入力エラーはローカルのみ**。  
- UPDATE 失敗・通信失敗時はエラーメッセージ。**編集中状態と入力値の扱い**は統一するとよい（例：失敗時は編集状態のまま内容を残す）。

### 4.5 メモ削除機能

| 項目 | 内容 |
|------|------|
| **目的** | 指定したメモの行を **データベースから削除**する |
| **Supabaseでの操作** | **DELETE** — 条件: **`id` = 対象メモの UUID** かつ **`user_id` = ログイン中ユーザー ID**。誤削除を防ぐため **確認ステップは必須** |

**処理の流れ**

1. ユーザーが **削除ボタン** を押す  
2. **確認ダイアログ** で「削除してよいか」を確認する  
3. キャンセルなら何もしない  
4. OK なら **DELETE** を実行する  
5. 成功したら：**一覧再読込**、**削除成功メッセージ**。編集中だったメモを消した場合は **編集状態の整合**（同じ ID を編集中なら解除・入力クリアなど）  

| 入力値 | 内容 |
|--------|------|
| 削除対象 id | `brain_dumps.id` |

| 出力・表示内容 | 内容 |
|----------------|------|
| 一覧 | 該当行が消える（再読込後） |
| メッセージ | 例：「削除しました」 |

**正常時の動き**

- 該当行が削除され一覧から消える  

**エラー時の動き**

- メッセージで失敗を表示。**一覧は古い状態のまま**でもよいので、ユーザーがページを再読み込みできる旨を伝えてよい  

---

### 4.6 メッセージ表示機能

| 項目 | 内容 |
|------|------|
| **目的** | 処理結果やエラー原因を **一箇所（メッセージ表示エリア）** に簡潔に示す |
| **Supabaseでの操作** | なし（表示専用） |

**表示する種類と文言の例（実装時に調整可）**

| 種類 | 表示タイミング | 出力・表示内容の例 |
|------|----------------|--------------------|
| 保存成功 | 新規 INSERT 成功後 | 「保存しました」 |
| 更新成功 | UPDATE 成功後 | 「更新しました」 |
| 削除成功 | DELETE 成功後 | 「削除しました」 |
| 入力エラー | 必須未入力など | 「タイトルと本文を入力してください」など |
| Supabase接続エラー／通信エラー | ネットワーク、タイムアウトなど | 「接続に失敗しました。時間をおいて再度お試しください」など |
| その他エラー | SDK が返したエラー等で上記以外 | 「処理に失敗しました」など **汎用文** とし、開発者コンソールで詳細確認は実装側 |

**処理の流れ（共通）**

1. メッセージ種別が決まる  
2. **メッセージ表示エリア** に文言をセットする  
3. **自動で消える／次の処理で置き換わる**など、画面上の見通しが良くなる運用とする  

**エラー時の動き**

- メッセージが読めない状態（見切れ）は避ける。可能なら色やアイコンで成功と失敗を分けてよい (**デザインは実装側**)。

### 4.7 新規登録機能

| 項目 | 内容 |
|------|------|
| **目的** | メールアドレス・パスワードでアカウントを作成し、メモ画面へ進める |
| **Supabaseでの操作** | **Auth `signUp`** — `email`, `password` を送信。**メール確認は行わない**（プロジェクト設定で Confirm email を無効化する前提） |
| **画面** | `index.html`（認証 UI 表示中） |

**処理の流れ**

1. ユーザーがメールアドレス・パスワード（6文字以上）を入力し **新規登録** を実行する  
2. 入力チェック：メール形式の簡易チェック（空不可）、パスワード **6 文字未満なら `alert()` で終了**（Supabase は呼ばない）  
3. **signUp** を実行する  
4. 成功したら **同一 `index.html` 内でメモ UI を表示**する（セッションが得られない構成の場合はログインを促す文言を `alert()` してもよいが、本要件では confirm email 無効のためセッション取得を想定）  
5. 失敗時は **`alert()`** にエラー要旨を表示する

### 4.8 ログイン機能

| 項目 | 内容 |
|------|------|
| **目的** | 既存アカウントでサインインし、メモ画面へ進める |
| **Supabaseでの操作** | **Auth `signInWithPassword`** — `email`, `password` |
| **画面** | `index.html`（認証 UI 表示中） |

**処理の流れ**

1. ユーザーがメールアドレス・パスワードを入力し **ログイン** を実行する  
2. 入力チェック：空不可、パスワード **6 文字未満なら `alert()`**  
3. **signInWithPassword** を実行する  
4. 成功したら **同一 `index.html` 内でメモ UI を表示**  
5. 失敗時は **`alert()`**（例：認証情報が正しくない）  

**補足：** ページ表示時に既にセッションがあれば、**認証 UI は出さずメモ UI を表示**する。

---

### 4.9 ログアウト機能

| 項目 | 内容 |
|------|------|
| **目的** | 認証状態を終了し、再ログインを促す |
| **Supabaseでの操作** | **Auth `signOut`** |
| **画面** | `index.html`（ログアウトボタン） |

**処理の流れ**

1. ユーザーが **ログアウト** を押す  
2. **signOut** を実行する  
3. 成功後 **`index.html` 上で認証 UI を表示**し、メモ UI を非表示にする  
4. 失敗時は **`alert()`**  

---

### 4.10 ログイン中メール表示機能

| 項目 | 内容 |
|------|------|
| **目的** | どのアカウントで操作しているか分かるようにする |
| **Supabaseでの操作** | **Auth `getSession` / `getUser`** 等でセッションから `email` を取得（表示専用） |
| **画面** | `index.html`（ヘッダー付近の表示領域） |

**処理の流れ**

1. **`index.html` 表示時**にセッションを確認する。無効なら **認証 UI のみ表示**し、メモ機能は起動しない  
2. 有効なら **ユーザーのメールアドレス**を画面上に表示する  
3. ログアウト後は表示しない

## 5. 機能間の関連（一覧）

| タイミング | 動く機能 |
|------------|-----------|
| `index.html` 表示直後 | セッション確認（あればメモ UI、なければ認証 UI） |
| 新規登録成功 | 新規登録機能 → メモ UI 表示・一覧読込 |
| ログイン成功 | ログイン機能 → メモ UI 表示・一覧読込 |
| メモ UI 表示中 | ログイン中メール表示機能 → カテゴリ選択肢読込・メモ一覧表示 |
| ログアウト | ログアウト機能 → 認証 UI 表示 |
| 保存ボタン（新規） | メモ新規保存機能 → メッセージ表示機能・一覧表示機能 |
| 保存ボタン（編集中） | メモ編集機能 → メッセージ表示機能・一覧表示機能 |
| 削除 OK 確定後 | メモ削除機能 → メッセージ表示機能・一覧表示機能 |
