# ブレインダンプアプリ 機能仕様書（バニラHTML + Supabase）

## 1. 文書概要

| 項目 | 内容 |
|------|------|
| アプリ名 | ブレインダンプ |
| データ保存先 | Supabase PostgreSQL、`public.brain_dumps`（任意で `memo_categories` に FK）および **`public.memo_categories`**（マスタ） |
| クライアント | `index.html` 1ファイル（HTML / CSS / JavaScriptのみ）。npmなし。**Supabase JavaScript SDK は CDN で読み込む**。 |

**関連仕様：** テーブル・カラムの定義は **`docs/02_DB仕様書.md`** を参照。  
本書では **画面上の機能と処理フロー、Supabase 操作の対応** を定義する。**コード・SQL・Supabase の URL／anon key 等は本書には書かない。**

---

## 2. 機能一覧

| No | 機能名 | 目的（一言） |
|----|--------|----------------|
| 1 | カテゴリ選択肢読込機能 | メモ入力用ドロップダウンを **`memo_categories` の内容** で埋める（`sort_order` 昇順など DB 仕様に合わせる） |
| 2 | メモ新規保存機能 | 入力したタイトル・本文および**任意の**カテゴリをデータベースに新規追加する |
| 3 | メモ一覧表示機能 | 保存済みメモを新しい順に画面に並べ、**カテゴリ名**も分かるように見せる |
| 4 | メモ編集機能 | 既存メモ（タイトル・本文・**カテゴリ任意**）を修正しデータベースに反映する |
| 5 | メモ削除機能 | 不要なメモをデータベースから削除する |
| 6 | メッセージ表示機能 | 成功・検証エラー・通信エラー等をユーザーに伝える |

---

## 3. 共通の前提・用語

| 用語 | 説明 |
|------|------|
| **Supabaseクライアント** | CDN から読み込んだ SDK で初期化し、`brain_dumps` および `memo_categories` にクエリするオブジェクト。**接続情報の設定は実装側で別途。** |
| **編集中フラグ／編集中 ID** | 編集対象メモの `id`（UUID）をブラウザ内で保持している状態。無ければ「新規」、あれば「更新」。 |
| **一覧の並び順** | `created_at` の **降順**（新しいものが先）。一覧表示機能と共通。 |

**入力チェックの共通ルール（推奨）：**

- **タイトル・本文とも必須**（DB が NOT NULL に合わせる）。
- 送信前に **前後の空白を除去（trim）** したうえで、空になったらエラーとしてメッセージ表示し **Supabase へは送信しない**。
- **カテゴリは任意**。ドロップダウン未選択時は **`category_id` を NULL** として送信し、一覧ではカテゴリ名なし（または占位表示）でよい。

長さの上限などは DB 側で明示的な制限が無い場合、**異常に大きな入力が遅くなる**点だけ留意（今回仕様での数値上限は別途未定でよい）。

## 4. 機能別仕様

### 4.1 カテゴリ選択肢読込機能

| 項目 | 内容 |
|------|------|
| **目的** | 投稿／編集フォームのドロップダウン選択肢を、**DB の `memo_categories`** で構成する |
| **Supabaseでの操作** | **SELECT** — `memo_categories` から **`id`, `name`（および並び順用の `sort_order`）** を取得し、**`sort_order` 昇順**など DB 仕様に従って並べる |

**処理の流れ**

1. **ページ表示後**に（一覧取得とは独立して実行してもよいが、入力前に済ませるとよい）  
2. `memo_categories` を **SELECT** する  
3. 取得結果を **ドロップダウンの option** として並べる。先頭に **値なし＝「未選択」相当** を置き、ユーザーが選ばなかった場合 **`category_id` は NULL** として保存する  

**正常時**

- メモの保存機能はカテゴリ未選択でも利用できること（要求仕様 FR-07）。  

**エラー時**

| 種別 | 推奨挙動 |
|------|----------|
| 読み込み失敗 | メッセージで通知。ドロップダウンを無効化するか、そのままでも **メモ一覧は可能なら表示** とするかは実装で統一するとよい |

---

### 4.2 メモ新規保存機能

| 項目 | 内容 |
|------|------|
| **目的** | ユーザーが入力した内容を **`brain_dumps` に新しい行として追加**し、一覧に反映させる |
| **Supabaseでの操作** | **INSERT（行の追加）** — `brain_dumps` に `title`, `content` を指定。**`category_id`** はユーザーが選択した場合のみセットし、未選択なら **`NULL`**（または同等の送信）。**`id` と `created_at` はサーバ側既定でセット**される想定（DB仕様に合わせる） |

**処理の流れ（正常系の目安）**

1. ユーザーがタイトルと本文を入力する（カテゴリは任意）
2. **編集中 ID が無いこと** を確認する（または保存時に無視して新規扱いになるよう統一する）  
3. ユーザーが **保存ボタン** を押す  
4. **入力チェック**（空白のみ不可）。NGなら メッセージ表示機能 でエラー文言を表示し終了  
5. Supabase に **INSERT** を実行する（`title`, `content`, および任意の `category_id`）
6. 成功したら：**入力欄を空にする**（カテゴリも未選択に戻す）、**編集中状態を解除**（念のため）、**一覧を再読込**する、成功メッセージを表示する  

| 入力値 | 内容 |
|--------|------|
| タイトル | 画面上のタイトル入力欄 → DB の `title` |
| 本文 | 画面上の本文入力欄 → DB の `content` |
| カテゴリ | 選択中の **`memo_categories.id`** → DB の `category_id`。未選択なら **`NULL`** |

| 出力・表示内容 | 内容 |
|----------------|------|
| 画面上のリスト | INSERT 後、一覧読み込みで **追加された行が先頭付近に表示**される（新しい順） |
| メッセージ | 例：「保存しました」など成功メッセージ |

**正常時の動き**

- データが1行増える  
- フォームが空になる  
- 一覧が最新になる  
- 成功メッセージが見える  

**エラー時の動き**

| 種別 | ユーザーへの伝え方 | アプリ側の推奨挙動 |
|------|--------------------|---------------------|
| 入力エラー（空など） | 「タイトルと本文を入力してください」等 | Supabase は呼ばない。入力内容は温存 |
| Supabase がエラーを返した | メッセージ表示エリアに **簡潔な失敗文言**（技術細語は多用しない） | 入力内容は温存。**再試行しやすい**状態にする |
| ネットワーク断など | 「接続に失敗しました」等 | 入力内容は温存 |

### 4.3 メモ一覧表示機能

| 項目 | 内容 |
|------|------|
| **目的** | 起動時および更新後に、**保存済みメモを一覧**で見せ、**カテゴリ名**が分かるようにする |
| **Supabaseでの操作** | **SELECT** — `brain_dumps` を **`created_at` 降順**で取得。リレーション（例：**`memo_categories` の `name` を埋め込み取得**／LEFT JOIN と等価）で **カテゴリ表示名を同時に得る**。**必要なカラム：** `id`, `title`, `content`, `created_at`, `category_id`, および表示用 **`memo_categories.name`**（名前は FK 側の取得でよい）。 |

**処理の流れ**

1. **画面を開いたとき**（および保存・更新・削除の成功後に「再読み込み」として同じ処理を実行）  
2. Supabase に一覧用の **SELECT** を実行する  
3. 取得結果を画面上の **メモ一覧エリア** に描画する（**カテゴリ未設定**は名前なしや「―」などで表示）  
4. 各メモに **編集ボタン** と **削除ボタン** を付ける  

| 入力値 | 内容 |
|--------|------|
| なし | 画面上の入力値は一覧取得には不要。**フィルタ・検索条件も今回つけない** |

| 出力・表示内容 | 内容 |
|----------------|------|
| 一覧項目 | **タイトル、本文の要約または冒頭、`created_at`、カテゴリ名**。各行に編集／削除。**`id` は画面に出さなくてもよい**が操作に必要 |

**正常時の動き**

- メモが0件なら一覧は空または「メモがありません」等の案内でもよい（文言は実装で調整）。  
- 1件以上なら指定順で表示。  

**エラー時の動き**

- メッセージ表示機能で「一覧を読み込めませんでした」等。**可能なら**再読込の手段（ページ再読み込みのみでも可）を示す。

### 4.4 メモ編集機能

| 項目 | 内容 |
|------|------|
| **目的** | 既存メモの `title` / `content`／**任意の `category_id`** を変えて **同じ `id` の行を更新**する |
| **Supabaseでの操作** | **UPDATE（行の更新）** — 対象は **`id` が編集中として保持している UUID** と一致する行。**更新項目は `title`, `content`, `category_id`（※未選択時は `NULL`）** など DB 実装と一致させる。**DB に `updated_at` が無い前提でもよい**。 |

**処理の流れ**

1. ユーザーが某一覧項目の **編集ボタン** を押す  
2. そのメモの **`id`** を編集中として保持する  
3. サーバーから当該行を **SELECT（または一覧キャッシュから）**して **`title`, `content`, `category_id`** を **タイトル・本文入力欄・カテゴリ選択** に表示する  
4. ユーザーが内容を変更し **保存ボタン** を押す  
5. **入力チェック**。NGならメッセージのみ  
6. **編集中 ID** をキーに **UPDATE** を実行する（`title`, `content`, **`category_id`（任意、`NULL` 可）**）  
7. 成功したら：**編集中状態解除**、**入力欄を空にする**（カテゴリも未選択に戻す）、**一覧再読込**、成功メッセージ  

| 入力値 | 内容 |
|--------|------|
| 編集対象の id | `brain_dumps.id` |
| 更新後タイトル | `title` |
| 更新後本文 | `content` |
| カテゴリ | 選択中マスタ **`id`** → `category_id`、未選択なら **`NULL`** |

| 出力・表示内容 | 内容 |
|----------------|------|
| 一覧 | UPDATE 後に再取得した内容が反映されていること |

**正常時の動き**

- データベース上の該当行のみ内容が変わる（`created_at` はそのままでよい。**DB仕様上 `updated_at` は任意**）。  

**エラー時の動き**

- INSERT と同様、**入力エラーはローカルのみ**。  
- UPDATE 失敗・通信失敗時はエラーメッセージ。**編集中状態と入力値の扱い**は統一するとよい（例：失敗時は編集状態のまま内容を残す）。  

---

### 4.5 メモ削除機能

| 項目 | 内容 |
|------|------|
| **目的** | 指定したメモの行を **データベースから削除**する |
| **Supabaseでの操作** | **DELETE（行の削除）** — 条件：**`id` が対象メモの UUID**。誤削除を防ぐため **確認ステップは必須** |

**処理の流れ**

1. ユーザーが **削除ボタン** を押す  
2. **確認ダイアログ** で「削除してよいか」を確認する  
3. キャンセルなら何もしない  
4. OK なら **DELETE** を実行する  
5. 成功したら：**一覧再読込**、**削除成功メッセージ**。編集中だったメモを消した場合は **編集状態の整合**（同じ ID を編集中なら解除・入力クリアなど）  

| 入力値 | 内容 |
|--------|------|
| 削除対象 id | `brain_dumps.id` |

| 出力・表示内容 | 内容 |
|----------------|------|
| 一覧 | 該当行が消える（再読込後） |
| メッセージ | 例：「削除しました」 |

**正常時の動き**

- 該当行が削除され一覧から消える  

**エラー時の動き**

- メッセージで失敗を表示。**一覧は古い状態のまま**でもよいので、ユーザーがページを再読み込みできる旨を伝えてよい  

---

### 4.6 メッセージ表示機能

| 項目 | 内容 |
|------|------|
| **目的** | 処理結果やエラー原因を **一箇所（メッセージ表示エリア）** に簡潔に示す |
| **Supabaseでの操作** | なし（表示専用） |

**表示する種類と文言の例（実装時に調整可）**

| 種類 | 表示タイミング | 出力・表示内容の例 |
|------|----------------|--------------------|
| 保存成功 | 新規 INSERT 成功後 | 「保存しました」 |
| 更新成功 | UPDATE 成功後 | 「更新しました」 |
| 削除成功 | DELETE 成功後 | 「削除しました」 |
| 入力エラー | 必須未入力など | 「タイトルと本文を入力してください」など |
| Supabase接続エラー／通信エラー | ネットワーク、タイムアウトなど | 「接続に失敗しました。時間をおいて再度お試しください」など |
| その他エラー | SDK が返したエラー等で上記以外 | 「処理に失敗しました」など **汎用文** とし、開発者コンソールで詳細確認は実装側 |

**処理の流れ（共通）**

1. メッセージ種別が決まる  
2. **メッセージ表示エリア** に文言をセットする  
3. **自動で消える／次の処理で置き換わる**など、画面上の見通しが良くなる運用とする  

**エラー時の動き**

- メッセージが読めない状態（見切れ）は避ける。可能なら色やアイコンで成功と失敗を分けてよい (**デザインは実装側**)。
